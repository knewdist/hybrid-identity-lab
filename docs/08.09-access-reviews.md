# Phase 08.9 — Access Reviews

## Overview

In this phase of the lab, Microsoft Entra **Access Reviews** are implemented to periodically validate membership in a privileged administrative group.

Access Reviews are part of **Microsoft Entra Identity Governance** and are used to ensure that users maintain access **only when it is still required**.

For this lab, the `Cloud-Admins` group is reviewed on a recurring basis by a governance team.

---

# Why Access Reviews Matter

Over time, users often retain access longer than necessary. This is known as **privilege creep**.

Access Reviews help prevent this by periodically requiring reviewers to confirm that users still need their assigned permissions.

Access Reviews support several core security principles:

- **Least Privilege**
- **Zero Trust**
- **Access Certification**
- **Regulatory Compliance**

---

# Lab Design

## Reviewed Resource

The group reviewed in this lab:


Cloud-Admins


This group represents **privileged administrative access** within the cloud environment.

---

## Reviewers

Instead of assigning a single user as the reviewer, a governance group is used.


Cloud-Governance


Using a group ensures that the review process continues functioning even if individual administrators change roles or leave the organization.

---

## Review Configuration

The Access Review was configured with the following settings:

| Setting | Value |
|------|------|
| Scope | All users |
| Recurrence | Quarterly |
| Duration | 7 days |
| Auto apply results | Enabled |
| If reviewers do not respond | Remove access |

This configuration ensures that privileged access is reviewed regularly and automatically enforced.

---

# Governance Workflow

The access governance model implemented in this lab follows this workflow:


User added to Cloud-Admins
↓
Quarterly access review triggered
↓
Cloud-Governance reviewers evaluate membership
↓
Approve or remove access
↓
System automatically enforces results


This separates **privileged access** from **governance oversight**, which reflects real-world enterprise IAM architecture.

---

# Configuration Steps

## Step 1 — Create the Access Review

In the **Microsoft Entra Admin Center**:

1. Navigate to **Identity Governance**
2. Select **Access Reviews**
3. Click **Create review**
4. Choose **Teams + Groups**
5. Select **Cloud-Admins**
6. Set the review scope to **All users**

Screenshot:

![Create Access Review](../screenshots/08.9/review-scope.png)

---

## Step 2 — Configure Reviewers and Recurrence

The review was configured to use a governance group.

Configuration used in this lab:

- **Reviewer type:** Selected users or groups
- **Reviewer group:** Cloud-Governance
- **Review recurrence:** Quarterly
- **Review duration:** 7 days
- **End condition:** Never

Using a governance group prevents the review process from depending on a single administrator.

Screenshot:

![Reviewers and Recurrence](../screenshots/08.9/reviewers-recurrence.png)

---

## Step 3 — Configure Completion Settings

The review was configured to automatically enforce governance decisions.

Settings applied:

- **Auto apply results to resource:** Enabled
- **If reviewers don't respond:** Remove access
- **No sign-in within 30 days:** Enabled
- **Justification required:** Enabled
- **Email notifications:** Enabled
- **Reminders:** Enabled

These settings simulate a **security-first governance model** where privileged access is automatically removed if it is not reviewed.

# Design Decisions

## Why review the `Cloud-Admins` group?

Reviewing a **privileged group** is more representative of real-world identity governance than reviewing a general user group.

Privileged groups typically require **more frequent and strict access validation**.

---

## Why use the `Cloud-Governance` group as reviewers?

Assigning the review to a governance group:

- avoids single-person dependency
- allows multiple security administrators to participate
- reflects real enterprise IAM governance teams

---

## Why quarterly reviews?

Quarterly reviews are commonly used for privileged access certification because they balance:

- strong security oversight
- manageable operational workload

---

## Why remove access if reviewers do not respond?

This configuration implements a **fail-secure model**:


No review decision
↓
Access removed


This prevents privileged access from remaining indefinitely without verification.

# Key Takeaway

Identity security is not only about authentication and conditional access.

Organizations must also **govern who retains access over time**.

Access Reviews ensure that privileged group membership remains:

- intentional
- justified
- regularly validated
