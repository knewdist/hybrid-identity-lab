# 08.7 — Risk-Based Conditional Access (Require MFA for Risky Sign-ins)

## Objective

Implement a **risk-adaptive Conditional Access policy** that requires Multi-Factor Authentication (MFA) when Microsoft Entra detects a risky sign-in.

This demonstrates how **Identity Protection integrates with Conditional Access** to enforce security controls dynamically based on risk signals.

This is a core **Zero Trust identity security control**.

---

# Environment

Tenant:

simmonslab.onmicrosoft.com

Identity platform:

Microsoft Entra ID

Security features used:

* Conditional Access
* Identity Protection
* Multi-Factor Authentication
* Privileged Identity Management (PIM)

---

# Background

Microsoft Entra **Identity Protection** continuously evaluates authentication attempts and assigns risk levels based on telemetry and machine learning.

Examples of risk signals include:

* Impossible travel
* Suspicious IP addresses
* Anonymous proxy usage
* Malware-linked infrastructure
* Credential leak detections

These risk signals can trigger **Conditional Access policies**.

Authentication decision flow:

```
User login
      │
      ▼
Microsoft Entra authentication
      │
      ▼
Identity Protection risk evaluation
      │
      ▼
Conditional Access policy evaluation
      │
      ▼
MFA required if risk detected
```

---

# Step 1 — Navigate to Conditional Access

Open:

```
Microsoft Entra ID
↓
Protection
↓
Conditional Access
↓
New Policy
```

---

# Step 2 — Create Policy

Policy Name:

```
Require MFA for Risky Sign-ins
```

This policy enforces MFA when Microsoft Entra detects suspicious login activity.

---

# Step 3 — Assign Users

Assignments:

```
Users or agents
↓
All users
```

In enterprise environments it is common to **exclude break-glass emergency accounts**, but this lab applies the policy to all users.

---

# Step 4 — Target Resources

Target resources:

```
All resources
(formerly "All cloud apps")
```

This ensures the policy applies across the Microsoft cloud environment.

---

# Step 5 — Configure Risk Condition

Conditions:

```
Sign-in risk
```

Configuration:

```
Medium
High
```

Meaning:

```
If Microsoft detects suspicious login behavior
↓
Trigger Conditional Access policy
```

---

# Step 6 — Configure Access Control

Access Controls:

```
Grant access
↓
Require multi-factor authentication
```

This ensures the user must complete an MFA challenge before access is granted.

---

# Step 7 — Enable Policy

Policy state:

```
Enabled
```

This activates the risk-based authentication control.

---

# Result

Authentication now follows this workflow:

```
User login attempt
        │
        ▼
Microsoft Entra authentication
        │
        ▼
Identity Protection evaluates risk
        │
        ▼
Risk detected (Medium or High)
        │
        ▼
Conditional Access policy triggered
        │
        ▼
MFA required
        │
        ▼
Access granted
```

This enables **adaptive authentication**.

---

# Security Impact

This policy helps protect against:

* Credential stuffing attacks
* Password spraying
* Phishing-based account compromise
* Suspicious login infrastructure
* Stolen credentials

Instead of requiring MFA for every login, the system enforces **risk-based authentication**.
